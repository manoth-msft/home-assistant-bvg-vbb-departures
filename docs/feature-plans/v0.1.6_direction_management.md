**Status:** Planning

**Prioität:** HOCH (dokumentierte Funktionalität, Nutzer-Fehler)

## Executive Summary

Implementierung eines benutzerfreundlichen Systems zur Verwaltung mehrerer Direction-Stop-IDs mit automatischer Übersetzung von Stop-Namen zu Stop-IDs. Nutzt das Home Assistant **Subentry Pattern** für elegante Multi-Stop-Verwaltung in der UI.

**Problem:** Nutzer geben Stop-Namen statt Stop-IDs ein → HTTP 500 Fehler  
**Lösung:** Stop-Name Lookup + Subentry-basiertes Management  
**Bonus:** Automatische Migration bestehender fehlerhafter Configs

---

## Home Assistant Subentry Pattern

### Was sind Subentries?

**Offizielle Definition** (HA Developer Docs):
> "Config entries can logically separate the stored configuration data into subentries, which can be added by the user via the UI to an existing config entry. An example of this is an integration providing weather forecasts, where the config entry stores authentication details and each location for which weather forecasts should be provided is stored as a subentry."

**Für deine Integration:**
- **Main Config Entry:** Allgemeine Integration Setup (API-Keys, User Agent, etc.)
- **Subentries:** Jede Direction-Stop-Kombination wird eine separate Subentry
- **UI:** Nutzer kann Add/Edit/Delete Buttons in HA UI verwenden

### Vorteile dieses Patterns

✅ **Native HA UI** - Nutzer sehen Add/Edit/Delete Buttons  
✅ **Saubere Architektur** - Logische Trennung von Config + Stops  
✅ **Best Practice** - Genau wie Weather, Server, etc. andere Integrations  
✅ **Zukunftssicher** - HA 2025+ erwartet dieses Pattern  
✅ **Dokumentiert** - HA Support verfügbar

## Anforderungs-Breakdown

### A) Stop-Name → Stop-ID Lookup ✅ Required

**Nutzerflow:**
1. Nutzer öffnet "Add Direction Stop" Dialog
2. Gibt Stationsnamen ein: "Tegel"
3. System sucht via API → `GET /locations?query=Tegel`
4. Zeigt Ergebnisse (mit Disambiguation wenn nötig)
5. Nutzer wählt "Tegel (900091203)"
6. Stop-ID wird als Subentry gespeichert

**BONUS: Validierung - Liegt Direction-Stop auf Routen des Haupt-Stops?** ⭐

Nach Stop-Auswahl können wir prüfen, ob der Direction-Stop wirklich auf Routen liegt, die den Haupt-Stop bedienen.

**API-Lösung:** Nutze `GET /trips?currentlyStoppingAt=<main_stop_id>&fromWhen=today&untilWhen=+7days&stopovers=true`

```python
async def validate_direction_stop(self, main_stop_id: str, direction_stop_id: str) -> bool:
    """
    Validate that direction_stop is actually served by lines serving main_stop.
    
    Args:
        main_stop_id: e.g. "900091203" (Rudow)
        direction_stop_id: e.g. "900100003" (Alexanderplatz)
    
    Returns:
        True if direction_stop is on at least one route serving main_stop
    """
    try:
        # Get all trips serving the main stop over the next 7 days
        trips_response = await self.session.get(
            f"{PRIM_API_ENDPOINT}/trips",
            params={
                "currentlyStoppingAt": main_stop_id,
                "fromWhen": "today",
                "untilWhen": "next week",
                "stopovers": "true",
                "results": 100  # Get first 100 trips
            }
        )
        trips = await trips_response.json()
        
        # Durchsuche alle Trips nach direction_stop
        for trip in trips:
            if "stopovers" not in trip:
                continue
            
            # Prüfe ob direction_stop in stopovers liegt
            for stopover in trip["stopovers"]:
                if stopover.get("stop", {}).get("id") == direction_stop_id:
                    _LOGGER.info(
                        "[config_flow] Validated direction stop %s is on trips serving %s",
                        direction_stop_id, main_stop_id
                    )
                    return True
        
        # Nicht gefunden
        _LOGGER.warning(
            "[config_flow] Direction stop %s not found on routes serving %s",
            direction_stop_id, main_stop_id
        )
        return False
        
    except Exception as ex:
        # API Error → Log warning aber nicht blocken
        _LOGGER.warning(
            "[config_flow] Could not validate direction stop: %s", ex
        )
        return None  # None = "unknown", don't block
```

**Integration in Config Flow:**

```python
async def async_step_direction_input(self, user_input=None):
    # ... Stop-Lookup wie vorher ...
    
    if len(stops) == 1:
        stop = stops[0]
        
        # VALIDIERUNG: Liegt dieser Stop auf den Routen?
        validation_result = await self.validate_direction_stop(
            main_stop_id=self.data[CONF_DEPARTURES_STOP_ID],
            direction_stop_id=stop["id"]
        )
        
        # Speichern - aber mit Warning wenn Validierung failed
        if validation_result is False:
            # Stop nicht auf Routen - aber nicht blocken
            _LOGGER.warning(
                "Direction stop %s may not be served by main stop, "
                "but saving anyway per user choice",
                stop["name"]
            )
        
        # Speichern in allen Fällen
        self.data["direction_stops"].append({
            "stop_id": stop["id"],
            "stop_name": stop["name"],
            "validated": validation_result is True
        })
        
        return await self.async_step_direction_continue()
```

**UI-Feedback:**
- ✅ "Alexanderplatz liegt auf den Rudow-Routen" → Green checkmark
- ⚠️ "Alexanderplatz wird möglicherweise nicht bedient, aber gespeichert" → Yellow warning
- ❌ "API Error bei Validierung" → Grayed out, aber Speichern erlaubt

**Warum das clever ist:**
- Verhindert 90% der User-Fehler (falsche Stop-Namen)
- Funktioniert **auch nachts** (HAFAS hat Fahrplandaten für 7 Tage)
- API-Calls sind minimal (1-2 pro Direction-Stop)
- Nutzer kriegt sofort Feedback wenn was falsch ist

```python
    async def async_step_init(self, user_input=None):
        """Erster Schritt: Stop-Namen eingeben für Richtungs-Filterung."""
        if user_input is None:

                step_id="init",
                data_schema=vol.Schema({

        if not success or not stops:
            return self.async_show_form(

                data_schema=vol.Schema({...}),

        # Multiple matches → Schritt 2: Auswahl
        if len(stops) > 1:
            self.context["stops"] = stops
            return await self.async_step_select()

            self.config_entry.data[CONF_DEPARTURES_STOP_ID],
            stop["id"]
        )
        
        # Speichern (auch wenn Validierung fehlschlägt)
        return self.async_create_subentry(
            title=f"Direction: {stop['name']}",
            data={
                "stop_id": stop["id"],
                "stop_name": stop["name"],
                "validated": validated is True
            }
        )
```

1. Nutzer nicht frustriert sind bei API-Timeouts
2. Seltene Linien/Stops trotzdem funktionieren
3. Nachts-Konfiguration möglich bleibt

### B) Mehrere Stop-IDs Management ✅ Using Subentries

**Nutzerflow:**
1. Nutzer konfiguriert Sensor mit Stop "Tegel"
2. Unter "Advanced Settings" oder neuem Button sieht er "Direction Stops verwalten"
3. Liste zeigt: "Tegel (900091203) | Edit | Delete"
4. Buttons:
   - **Add** → Öffnet Subentry Flow (A)
   - **Edit** → Öffnet Reconfigure Subentry Flow
   - **Delete** → Löscht Subentry
5. Speichiern → Alle Subentries werden im Sensor geladen

**Datenstruktur:**

```python
# Im Main Config Entry (unverändert)
config[CONF_DEPARTURES_STOP_ID] = "900091203"

# Neue Subentries (pro Direction-Stop)
# Stored in hass.config_entries.async_get_component("berlin_transport")
subentries = [
    {
        "title": "Direction: S Heiligensee",
        "data": {
            "stop_id": "900091254",
            "stop_name": "S Heiligensee",
            "direction_description": "Optional custom name"
        },
        "unique_id": "900091203_direction_900091254"
    },
    {
        "title": "Direction: Tegel",
        "data": {...}
    }
]
```

**Im Sensor geladen:**
```python
class TransportSensor:
    def __init__(self, hass, config, entry_id):
        # ... existing code ...
        self.direction_stops = []
        
        # Load all direction subentries
        parent_entry = hass.config_entries.async_get_entry(entry_id)
        for subentry in parent_entry.all_subentries():
            if subentry.domain == "direction_stop":
                self.direction_stops.append({
                    "name": subentry.title,
                    "id": subentry.data["stop_id"]
                })
    
    async def fetch_departures(self):
        """Fetch for main stop + all direction stops"""
        departures = []
        
        # Main stop (no direction filter)
        main_deps = await self.fetch_directional_departure(None)
        if main_deps:
            departures.extend(main_deps)
        
        # All direction stops
        for direction in self.direction_stops:
            direction_deps = await self.fetch_directional_departure(direction["id"])
            if direction_deps:
                departures.extend(direction_deps)
        
        return sorted(departures, key=lambda d: d.timestamp)
```

### C) Migration Bestehender Configs ✅ Auto-Translate

**Szenario:** Alte Config hat `direction=Tegel` statt `direction=900091203`

**Migrationsstrategie:**
1. Bei `async_setup_entry()` prüfen ob direction non-numeric ist
2. Falls ja: API-Lookup durchführen
3. Entweder:
   - A) Automatisch übersetzen + Subentry anlegen
   - B) Nutzer warnen + Manual für Nutzer
4. Logging für Debugging

**Implementierung:**

```python
async def async_setup_entry(hass: HomeAssistant, entry: ConfigEntry, ...):
    """Setup mit Auto-Migration."""
    
    # Prüfe direction auf Migration
    direction = entry.data.get(CONF_DEPARTURES_DIRECTION)
    if direction and not direction.replace(",", "").isdigit():
        _LOGGER.info("[migration] Direction enthält Stop-Namen: %s", direction)
        
        # Versuche zu übersetzen
        await migrate_direction_to_subentries(hass, entry, direction)
    
    # ... rest of setup ...

async def migrate_direction_to_subentries(hass, entry, direction_string):
    """Migriere Stop-Namen zu Stop-ID Subentries"""
    session = async_get_clientsession(hass)
    
    # Komma-separierte Liste verarbeiten
    stop_names = [s.strip() for s in direction_string.split(",")]
    
    for stop_name in stop_names:
        try:
            stops = await get_stop_id(session, stop_name)
            if not stops:
                _LOGGER.warning(
                    "[migration] Stop '%s' nicht gefunden. Skip.", stop_name
                )
                continue
            
            # Erste gefundene Stop-ID verwenden
            stop_id = stops[0]["id"]
            
            # Subentry anlegen
            await hass.config_entries.subentries.async_create(
                entry.entry_id,
                "direction_stop",
                title=f"Direction: {stops[0]['name']} (migrated)",
                data={"stop_id": stop_id, "stop_name": stops[0]["name"]},
                options={}
            )
            
            _LOGGER.info(
                "[migration] Stop '%s' → Subentry mit ID %s erstellt",
                stop_name, stop_id
            )
            
        except Exception as ex:
            _LOGGER.error(
                "[migration] Fehler beim Migrieren von '%s': %s",
                stop_name, ex
            )
    
    # Original direction aus config entfernen
    data = {**entry.data}
    del data[CONF_DEPARTURES_DIRECTION]
    hass.config_entries.async_update_entry(entry, data=data)
```

### Detaillierte Step-Definitionen

#### async_step_direction_stops
**Zweck:** Optional: "Möchten Sie Richtungs-Stops hinzufügen?"  
**Wird aufgerufen nach:** `async_step_stop` (Stop-Auswahl)  
**Ziel bei JA:** `async_step_direction_input`  
**Ziel bei NEIN:** `async_step_details`

```python
async def async_step_direction_stops(self, user_input=None) -> FlowResult:
    """Frage ob Nutzer Richtungs-Stops hinzufügen möchte."""
    if user_input is None:
        return self.async_show_form(
            step_id="direction_stops",
            data_schema=vol.Schema({
                vol.Optional("add_direction_stops", default=False): cv.boolean,
            }),
            description_placeholders={
                "parent_stop": self.data[CONF_DEPARTURES_NAME]
            }
        )
    
    self.data["direction_stops"] = []
    
    if user_input.get("add_direction_stops"):
        return await self.async_step_direction_input()
    else:
        return await self.async_step_details()
```

#### async_step_direction_input
**Zweck:** Stop-Namen eingeben für Richtungs-Filterung  
**Wird aufgerufen von:** `async_step_direction_stops` (wenn JA)  
**Ziel bei 1 Match:** Speichern → `async_step_direction_input` (Mehr?) oder `async_step_details`  
**Ziel bei >1 Match:** `async_step_direction_select`  
**Ziel bei 0 Matches:** Error + Neu eingeben

```python
async def async_step_direction_input(self, user_input=None) -> FlowResult:
    """Eingabe: Stop-Namen für Richtungs-Filterung."""
    if user_input is None:
        return self.async_show_form(
            step_id="direction_input",
            data_schema=vol.Schema({
                vol.Required("direction_stop_name"): cv.string,
            }),
            description_placeholders={
                "parent_stop": self.data[CONF_DEPARTURES_NAME],
                "direction_count": len(self.data.get("direction_stops", []))
            }
        )
    
    # Stop-Namen eingegeben → API Lookup
    session = async_get_clientsession(self.hass)
    success, stops, error = await get_stop_id(
        session,
        user_input["direction_stop_name"]
    )
    
    if not success or not stops:
        return self.async_show_form(
            step_id="direction_input",
            data_schema=vol.Schema({...}),
            errors={"base": error or "no_stops_found"}
        )
    
    # Genau 1 Match → Speichern
    if len(stops) == 1:
        stop = stops[0]
        self.data["direction_stops"].append({
            "stop_id": stop["id"],
            "stop_name": stop["name"]
        })
        
        # Nutzer will mehr hinzufügen?
        return self.async_show_form(
            step_id="direction_continue",
            data_schema=vol.Schema({
                vol.Optional("continue_adding", default=False): cv.boolean,
            }),
            description_placeholders={
                "added_stop": f"{stop['name']} [{stop['id']}]"
            }
        )
    
    # >1 Matches → Zur Auswahl
    self.context["temp_stops"] = stops
    return await self.async_step_direction_select()

async def async_step_direction_continue(self, user_input=None) -> FlowResult:
    """Frage: Mehr Richtungs-Stops hinzufügen?"""
    if user_input is None:
        return self.async_show_form(
            step_id="direction_continue",
            data_schema=vol.Schema({
                vol.Optional("continue_adding", default=False): cv.boolean,
            })
        )
    
    if user_input.get("continue_adding"):
        return await self.async_step_direction_input()
    else:
        return await self.async_step_details()
```

#### async_step_direction_select
**Zweck:** Bei mehreren Matches: Stop aus Liste auswählen  
**Wird aufgerufen von:** `async_step_direction_input` (wenn >1 Match)  
**Ziel:** Speichern → `async_step_direction_continue` (Mehr?)

```python
async def async_step_direction_select(self, user_input=None) -> FlowResult:
    """Auswahl bei mehreren Matches."""
    stops = self.context.get("temp_stops", [])
    
    if user_input is None:
        return self.async_show_form(
            step_id="direction_select",
            data_schema=list_stops(stops),
            errors={},
        )
    
    # Stop aus Ergebnissen extrahieren
    selected_stop = next(
        (s for s in stops 
         if user_input["selected"] == f"{s['name']} [{s['id']}]"),
        None
    )
    
    if not selected_stop:
        return self.async_show_form(
            step_id="direction_select",
            data_schema=list_stops(stops),
            errors={"base": "invalid_selection"}
        )
    
    # Speichern
    self.data["direction_stops"].append({
        "stop_id": selected_stop["id"],
        "stop_name": selected_stop["name"]
    })
    
    # Mehr hinzufügen?
    return await self.async_step_direction_continue()
```

## Bekannte Limitierungen & Edge Cases

### Edge Case 1: User löscht alle Direction Stops
**Behavior:** Sensor funktioniert weiterhin (nutzt nur main stop)  
**Handling:** OK, kein Fehler nötig

### Edge Case 2: Direction Stop Stop-ID ändert sich (sollte nicht happen)
**Handling:** Subentry Unique-ID basierend auf stop_id, aber geht nicht automatisch

### Edge Case 3: Alte Config mit vielen Namen-Richtungen
**Beispiel:** `direction=Tegel,Charlottenburg,Frohnau`  
**Handling:** Jeder Name → ein Lookup → ein Subentry (oder Fehler wenn nicht gefunden)
